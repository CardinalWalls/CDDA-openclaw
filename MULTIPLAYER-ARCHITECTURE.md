# CDDA Agent Play - Multiplayer Architecture

## 核心问题

1. **存档可以被伪造** - 玩家可以修改本地文件，声称自己有神装
2. **世界观一致性** - 多个玩家要在同一个世界玩，经济系统要统一
3. **角色迁移** - "神角色"怎么进入共享世界而不破坏平衡

---

## 架构设计

### 原则：服务端权威

```
所有游戏状态都在服务端
客户端只是"显示"和"输入"
客户端不能直接修改游戏状态
```

---

## 方案 A: 中心化服务端 (MMO 风格)

```
┌─────────────────────────────────────────┐
│           CDDA World Server               │
│  ├── 权威世界状态 (PostgreSQL)            │
│  ├── 玩家数据 (账户、角色)               │
│  ├── 物品/经济系统                       │
│  └── Agent AI 控制器                     │
└─────────────────────────────────────────┘
           ▲                    ▲
           │                    │
    ┌─────────────┐      ┌─────────────┐
    │  玩家 A      │      │  玩家 B     │
    │  CDDA 客户端  │      │  CDDA 客户端  │
    └─────────────┘      └─────────────┘
```

**问题：CDDA 是回合制单机游戏，改成 MMO 需要重写游戏核心，不现实。**

---

## 方案 B: 事件溯源 (Git 风格) - 推荐

```
核心思路: 不存最终状态，存所有"动作"的历史

传统存档:
  world.sav = 最终状态 (可伪造)

事件溯源:
  actions/
  ├── player_001/
  │   ├── action_001.json  # "移动到 x,y"
  │   ├── action_002.json  # "捡起 sword"
  │   └── action_003.json  # "攻击 zombie"
  └── player_002/
      ├── action_001.json
      └── ...

服务端重放:
  初始世界 + 所有 actions = 当前世界状态
```

**好处：**
- 每一动作都可验证
- 没法伪造，因为伪造的动作不在 action log 里
- 世界状态由所有动作重建，永远一致

## 经济系统设计

### 物品分类

```
可验证物品 (Verified):
  - 只能通过服务端生成
  - 生成时记录到 action log
  - 转移时服务端验证

本地物品 (Local):
  - 场景生成的 NPC/物品
  - 只在该世界有效
  - 不能带出这个世界
```

### 跨世界角色迁移

**场景：玩家有一个"神角色"，想带进公共世界**

```
问题：神角色破坏平衡

解决：角色降级 + 重新开始

1. 导出的角色进行分析
   - 统计各属性
   - 识别稀有物品
   - 计算"力量等级"

2. 降级规则
   - 力量等级 > 阈值 → 降级到阈值
   - 稀有物品 → 转换为"传承点数"
   - 传承点数可在新世界兑换基础装备

3. 新世界重新生成
   - 继承传承点数
   - 从零开始探索
```

## 防伪造机制

### 1. 存档签名

```
服务端:
  save_data = serialize(world_state)
  signature = sign(save_data, server_private_key)
  save_file = { save_data, signature }

客户端:
  - 只能读取
  - 不能写入未签名的存档
  - 载入前验证签名
```

### 2. 动作签名

```
每次动作:
  action = { player_id, action_type, params, timestamp }
  action_hash = hash(previous_action + action)
  action_signed = { action, action_hash, player_signature }
  
服务端:
  - 验证动作链完整
  - 验证签名
  - 拒绝无效动作
```

### 3. 实时验证

```
客户端发送动作 → 服务端验证 → 更新状态 → 广播结果

如果:
  - 动作不合法 → 拒绝
  - 签名无效 → 断开连接
  - 状态不一致 → 同步服务端状态
```

## Agent 怎么玩？

### 单人模式 (当前)

```
Agent ←→ 客户端 ←→ 本地 tmux ←→ CDDA

Agent 发送动作 → 客户端执行 → CDDA 响应
```

### 联机模式 (未来)

```
Agent ←→ 客户端 ←→ World Server ←→ CDDA Server Instance

Agent 发送动作 → 服务端验证 → 执行 → 广播给所有玩家
```

**注意：** CDDA 本身不支持真正的多人。需要：
1. CDDA Server (可能需要 mod 或改造)
2. 或者使用 JSON API 模式与 CDDA 交互
3. 或者运行多个 CDDA 实例同步状态

## 技术栈

```
服务端:
  - Go/Rust: 高性能网络服务
  - PostgreSQL: 世界状态存储
  - Redis: 实时缓存、会话
  - WebSocket: 客户端通信

客户端:
  - Tauri/Electron: GUI
  - WebSocket: 连接服务端
  - 本地 tmux: 运行 CDDA
```
