## 1. 冲突处理（类比 Git Rebase）

```
Git Worktree 场景:
  main branch = 共享世界
  worktree A = 玩家 A 的存档
  worktree B = 玩家 B 的存档
  
  A 和 B 都想把自己分支 rebase 到 main
  如果改了同一个文件 → 冲突
```

### CDDA 的冲突类型

```
类型 1: 地点冲突
  A: "我在 (100,200) 建造了 shelter"
  B: "我在 (100,200) 建造了 shelter"
  
  → 冲突：同一地点被重复使用

类型 2: 物品冲突
  A: "我捡到了 legendary sword"
  B: "我捡到了 legendary sword"
  
  → 冲突：唯一物品被多人声称拥有

类型 3: 事件冲突
  A: "我在 Day 10 杀了 zombie king"
  B: "我在 Day 10 杀了 zombie king"
  
  → 可能冲突：同一个 NPC 被多人击杀
```

### 冲突解决策略（像 Git）

```
Git 的三种冲突解决:
1. 接受 ours / theirs
2. 手动合并
3. 放弃其中之一

我们的对应策略:
```

### 策略 1: 时间优先 (ours)

```
规则: 先发生的事件优先

A 的事件: Day 5 建造 shelter
B 的事件: Day 6 建造 shelter

A 优先，B 的事件标记为 "relocated"

效果: 保留 A，B 移到附近空地
```

### 策略 2: 重要性优先

```
规则: 更"重要"的事件优先

判断重要性:
- 战斗 > 建造 > 移动
- 稀有物品发现 > 普通物品
- 成就解锁 > 一般动作

A: "杀了 zombie king" (重要性: 100)
B: "建了 shelter" (重要性: 50)

A 优先，B 的 shelter 保留但标注 "战区附近"
```

### 策略 3: 空间分离

```
规则: 如果冲突，同一地点的事件变成"邻近事件"

A 在 (100,200) 建 shelter
B 在 (100,200) 建 shelter

合并结果:
- A 的 shelter 在 (100,200)
- B 的 shelter 变成 (101,200) 或 (100,201)
- 标注: "B 的玩家在 A 附近建立了自己的基地"
```

### 策略 4: 玩家协商 (interactive rebase)

```
像 Git 的 interactive rebase:

共享世界服务器发现冲突
→ 暂停 B 的同步
→ 发消息给 B:
  "你的 shelter 位置和 A 冲突，选择:
   1. 移到旁边
   2. 合并成共享基地
   3. 放弃这个地点"
→ B 选择后，继续 rebase
```

## 2. 唯一稀有物品

CDDA 确实有唯一物品！

```
类型 1: 固定生成的神器 (Artifacts)
  - 位置固定
  - 生成时带 unique ID
  - 不能复制

类型 2: Bionic 植入物
  - 特定 NPC 独有
  - 通过任务获得

类型 3: 传说武器
  - 游戏内置的唯一武器
  - 如: "Locket of note" 等
```

### 唯一物品的哈希追踪

```json
{
  "unique_item": true,
  "item_id": "artifact_001",
  "spawn_event": "evt_042",
  "first_owner": "player_a",
  "theft_events": [],
  "current_owner": "player_b",
  "verified": true
}
```

### 唯一物品冲突解决

```
情况: A 和 B 都声称拥有同一 artifact

解决: 追溯哈希链
- 查 artifact_001 的 spawn_event
- 谁的事件记录更早/更完整？
- 早期事件优先

如果无法判断:
- artifact 归入"共享世界遗迹"
- 没人真正拥有
- 所有玩家都可以去争夺
```

## 3. 基础生产力设定

CDDA 有 world gen settings，是共享世界的基础！

```
CDDA World Gen 参数:

# 地图设置
MAPSIZE = 4          # 世界大小
SAVE_FMT = JSON     # 存档格式

# 生物密度
SPAWN_DENSITY = 0.5  # 僵尸密度
ITEM_DENSITY = 1.0  # 物品密度

# 时间
START_SEASON = spring
START_POINT = 0      # 起始季节

# 科技等级
START_DATE = 2020    # 末日日期
```

### 这些参数成为共享世界的"宪法"

```
共享世界的宪法:
{
  "world_seed": "shared_seed_001",
  "base_settings": {
    "SPAWN_DENSITY": 0.5,    // 所有玩家共享
    "ITEM_DENSITY": 1.0,      // 不能单方面修改
    "START_DATE": "2020-03-01",
    "START_SEASON": "spring"
  },
  "generated_features": {
    "cities": ["Rhyolith", "Conetoe"],  // 共享城市
    "special_locations": [...],
    "world_age": 14  // 游戏天数
  }
}
```

### 生产力的来源

```
共享世界的资源是有限的:

物品总量 = 世界生成时的总量 - 已被捡起的物品

如果 A 捡起了所有食物:
  → B 只能接受"这个世界食物很少"的现实
  → 或者通过交易从 A 那里获得

这创造了真实的经济!
```

## 4. 联机版架构

### 核心问题: 多人在同一个空间

```
传统 MMO: 玩家同时在线，实时同步

CDDA Agent Play: 
  - 回合制，每个玩家的 Agent 轮流行动
  - 玩家之间通过"回合同步"交互
```

### 回合制联机设计

```
┌─────────────────────────────────────────────────────┐
│              Turn Server (共享世界时钟)                  │
│                                                      │
│  world_turn = 0                                     │
│  players_online = [A, B, C]                         │
│                                                      │
│  turn_queue = [A, B, C, A, B, C, ...]              │
└─────────────────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────────┐
│  当轮到玩家 A:                                         │
│                                                      │
│  1. A 的 Agent 收到 "your turn"                      │
│  2. A 可以执行任意多动作                              │
│  3. A 说 "done"                                     │
│  4. world_turn++                                    │
│  5. 下一个玩家收到通知                               │
│                                                      │
│  其他玩家:                                           │
│  - 看到 A 的行动动画/日志                            │
│  - 等待自己的回合                                     │
└─────────────────────────────────────────────────────┘
```

### 玩家相遇时的交互

```
场景: A 和 B 在同一地图格子相遇

回合流程:
1. A 回合开始
   → A 看到 B 在格子里
   → A 选择: 交易 / 战斗 / 交谈 / 等待

2. A 选择"交易"
   → 打开交易面板
   → A 放入物品，B 放入物品
   → 双方确认
   → 交易完成

3. A 说 "done"
   → B 回合开始
   → B 看到 A 在面前
   → B 选择如何回应
```

### 共享世界的条件

```
世界要满足:

1. 确定性 (Deterministic)
   - 同一 seed = 同一世界
   - 所有玩家看到相同的地图

2. 实时同步 (Real-time Sync)
   - 玩家位置同步
   - 物品状态同步
   - 玩家动作同步

3. 回合顺序 (Turn Ordering)
   - 玩家列表
   - 当前轮到谁
   - 等待队列

4. 冲突解决 (Conflict Resolution)
   - 同时到达同一格子
   - 同时捡起同一物品
   - 谁的动作优先？
```

### 联机架构图

```
┌─────────────────────────────────────────────────────┐
│              CDDA Agent Play Server                    │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  World State (Authoritative)                    │  │
│  │  - 地图 tiles                                  │  │
│  │  - 玩家位置                                     │  │
│  │  - 物品位置                                     │  │
│  │  - NPC 状态                                     │  │
│  │  - 世界时间                                     │  │
│  └────────────────────────────────────────────────┘  │
│                        │                             │
│                        ▼                             │
│  ┌────────────────────────────────────────────────┐  │
│  │  Turn Manager                                   │  │
│  │  - turn_queue: [A, B, C]                       │  │
│  │  - current_turn: A                            │  │
│  │  - turn_timeout: 60s                          │  │
│  └────────────────────────────────────────────────┘  │
│                        │                             │
│                        ▼                             │
│  ┌────────────────────────────────────────────────┐  │
│  │  Sync Engine                                   │  │
│  │  - 广播玩家动作                                │  │
│  │  - 验证动作合法性                              │  │
│  │  - 处理冲突                                    │  │
│  └────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
              ▲                         ▲
              │                         │
    ┌───────────────┐         ┌───────────────┐
    │  玩家 A       │         │  玩家 B       │
    │  Agent 控制器 │         │  Agent 控制器 │
    └───────────────┘         └───────────────┘
```

### 冲突处理（同一格子）

```
情况: A 和 B 同时移动到 (100,200)

Turn Manager 的处理:

Option 1: 速度决定 (Speed-based)
  - 比较两个玩家的速度属性
  - 快的先到，慢的玩家看到对方已在

Option 2: 随机决定 (Coin flip)
  - 随机决定谁先到
  - 另一方收到 "格子已被占据" 消息

Option 3: 遭遇战 (Encounter)
  - 两人同时到达
  - 触发"遭遇"事件
  - 两人都可以行动，但需协商
```

## 总结

| 问题 | 解决方案 |
|------|----------|
| 冲突了怎么办 | 像 Git: 时间优先/重要性优先/空间分离/玩家协商 |
| 唯一稀有物品 | 哈希链追踪，第一次生成的优先 |
| 基础生产力 | World Gen 参数成为"宪法"，所有玩家共享 |
| 联机版架构 | 回合制 + Turn Server + 共享世界状态 |

---

## 关键技术细节

```
CDDA 回合制天然支持联机:

1. 回合推进:
   - 每个玩家行动消耗回合
   - 世界时间统一推进
   - 所有玩家看到相同时间

2. 地图同步:
   - CDDA 地图是确定性的
   - 同一 seed = 同一地图
   - 只需要同步玩家位置和物品变化

3. 物品系统:
   - 物品有 unique ID
   - 拾取/丢弃是原子操作
   - 冲突可通过 ID 追踪解决
```
