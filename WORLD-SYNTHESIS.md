# CDDA Agent Play - World Synthesis Architecture

## 核心理念

**世界不是预设的，是从玩家存档中生长出来的。**

每个玩家都有自己的游戏历程，这些历程汇聚成一个共享世界观。

---

## 问题 1: 存档伪造

**解决：存档内容签名 + 来源追踪**

```
存档 = { 事件列表, 签名 }

存档不能被伪造，因为：
1. 每个事件都有时间戳和哈希
2. 事件链可验证
3. 合成时验证所有存档的完整性
```

---

## 问题 2: 世界一致性

**解决：事件合成 (Event Synthesis)**

```
传统做法:
  服务端有"正确答案"，客户端同步

我们的做法:
  多个客户端的存档 → 提取事件 → 合成共享世界
  
每个存档都是"一个平行宇宙"
合成器把这些平行宇宙合并成一个"真实"世界
```

### 合成规则

```
1. 时间线对齐
   - 根据游戏内时间排序事件
   - 冲突时保留较早的（或者协商）

2. 地点解析
   - 同一地点的多个事件 → 合并描述
   - 冲突事件 → 保留影响更大的

3. 实体合并
   - 同名 NPC → 合并属性
   - 同名城市 → 合并状态
   - 物品 → 累加或按规则生成
```

---

## 问题 3: 角色迁移 ("神角色"问题)

**解决：世界锚定 + 实力平衡**

```
玩家想带"神角色"进共享世界？

1. 锚定角色历史
   - 角色的所有事件都在共享世界有记录
   - 其他玩家可以验证

2. 实力评估
   - 计算角色总实力 = 装备 + 属性 + 成就
   - 标记为"来自 solo 宇宙"

3. 平衡机制
   - 神角色进入共享世界后
   - 获得"旅行者" debuff (实力暂时封印)
   - 需要在新世界证明自己才能解锁
   - 或者分解神装，换成"传承点数"
```

## 冲突处理

### 冲突类型

```
1. 地点冲突
   - 玩家 A 说："我在 (100,200) 建了房子"
   - 玩家 B 说："我在 (100,200) 建了房子"
   
   解决：保留两者，标记为"邻居"
         或合并为"双城镇"

2. 时间冲突
   - 玩家 A："Day 10 我杀了 zombie"
   - 玩家 B："Day 10 我杀了 zombie"
   - 两个事件可能是同一个 zombie
   
   解决：保留两个事件，标记为"待验证"
         如果完全相同，合并

3. 实体冲突
   - 玩家 A 的 NPC "John" 是铁匠
   - 玩家 B 的 NPC "John" 是医生
   
   解决：按名称空间分离
         A_John 和 B_John
         或合并属性
```

## 角色迁移流程

```
1. 提交存档
   玩家上传自己的 CDDA 存档

2. 提取事件
   分析存档，生成事件链

3. 世界锚定
   检查事件与共享世界的兼容性
   - 地点不存在？→ 添加到世界
   - NPC 冲突？→ 合并或分离

4. 角色评估
   - 计算实力等级
   - 检查装备稀有度
   - 生成"旅行者档案"

5. 进入共享世界
   - 角色以"旅行者"身份出现
   - 实力被封印部分
   - 可以慢慢在新世界获得认可
```

## 经济系统

```
共享世界的资源是有限的：

1. 物品来源追踪
   - 每个物品都追溯到它的"出生事件"
   - 例如："sword_001" 由玩家 A 在 Day 5 锻造

2. 物品转移记录
   - 每次交易记录事件
   - 最终每个物品都有完整的"族谱"

3. 稀有度保证
   - "神装"不能凭空出现
   - 必须有合法的生成事件
   - 伪造的装备不会被共享世界接受
```

## 实现步骤

### Phase 1: 存档分析器
```
- 解析 CDDA 存档格式 (JSON)
- 提取玩家事件
- 生成事件哈希链
- 验证存档完整性
```

### Phase 2: 事件合成引擎
```
- 实现事件合并算法
- 处理地点冲突
- 生成共享时间线
```

### Phase 3: 原型验证
```
- 2-3 个玩家的存档合成测试
- 验证冲突处理
- 展示共享世界
```

### Phase 4: 联机模式
```
- 服务端部署
- 客户端对接
- 实时同步
```

## 关键技术

```
存档解析:
  - CDDA 使用 JSON 格式存档
  - 位置: ~/.cdda/saves/SAVE_NAME/
  - 文件: world_atx.json, scenarios.json, ...

事件提取:
  - 遍历玩家动作历史
  - 识别有意义的事件 (战斗、建造、交易、发现)
  - 生成标准化事件格式

哈希链:
  - 每个事件包含前一个事件的哈希
  - 类似区块链的思想
  - 保证事件链不可篡改
```
